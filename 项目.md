## 背景
这是我主导的一个金融级核心系统的云原生高可用架构升级与迁移项目。背景是，为了满足全球金融监管对极端灾备（RTO/RPO）的严苛要求，我们需要将数千个多租户计算节点，从传统的2可用区（2AZ）架构，平滑、零停机地迁移升级到3可用区（3AZ）的高可用架构。

## 项目一：金融云原生大数据平台重构与迁移方案

### ——基于 DataWorks + MaxCompute 的大数据平台建设与割接

这个项目的背景是，客户是一家全球性金融机构，原有数据体系比较分散，既有传统本地数据库，也有私有云上的数据平台，同时业务正在向云原生、大数据和 AIGC 场景演进。客户希望引入一套更标准化、更安全、可治理、可扩展的大数据平台，用于承载金融批处理、监管报表、风险分析、客户画像以及后续大模型微调和 RAG 应用的数据底座。

我在这个项目里的核心角色，是从解决方案角度参与前期调研、目标架构设计、迁移路径规划、数据校验方案设计以及割接风险控制。

### 1. 业务背景与引入原因

金融客户的大数据场景有几个明显特点：

第一，批处理占比较高。很多数据任务不是实时交易链路，而是 T+1、T+0、监管报送、风险统计、经营分析类任务，对稳定性、准确性、任务调度和失败重跑能力要求很高。

第二，安全和合规要求极高。金融数据通常涉及客户信息、交易数据、风控数据，不能只考虑“能跑起来”，还要考虑权限隔离、数据脱敏、审计追踪、访问控制和跨环境传输安全。

第三，上下游依赖复杂。大数据平台不是孤立系统，上游可能来自数据库、OSS、消息队列、业务系统，下游可能服务 BI 报表、风控模型、监管系统、AI 训练任务。因此迁移时不能只迁移表和任务，还要梳理依赖关系、调度链路和割接顺序。

基于这些特点，方案选择了阿里云 DataWorks + MaxCompute 作为核心组合。DataWorks 负责数据集成、开发、调度、运维、数据治理和监控；MaxCompute 负责大规模离线计算和存储承载。这个组合比较适合金融批处理场景，因为它能够把数据开发、任务编排、权限治理、资源隔离和成本控制统一起来。
```text
数据接入 → 数据开发 → 调度编排 → 权限治理 → 质量校验 → 运行监控

### 2. 核心痛点诊断

前期调研时，我们主要识别出几个问题。

第一，原有数据链路比较分散，任务依赖关系不清晰。很多任务之间存在上下游依赖，但缺乏统一的血缘分析和调度视图，导致迁移时不知道哪些任务可以先迁、哪些任务必须一起迁、哪些任务会影响下游报表或风控系统。

第二，数据迁移存在一致性风险。客户最关心的是金融数据不能错。早期如果只做表级别校验，比如行数、表大小、抽样比例，这在普通场景可能够用，但在金融场景不够，因为即使总行数一致，也可能存在字段值错误、分区数据缺失、主键重复或金额类字段精度问题。

第三，跨云、跨系统依赖复杂。除了大数据产品本身，还有数据库、对象存储、报表系统、调度系统、权限系统等非大数据产品依赖。两朵云之间需要保证网络互联互通，迁移期间还要保证旧平台和新平台可以并行运行。

第四，资源和成本缺乏精细化治理。原平台存在部分任务资源申请过大、低峰资源闲置、高峰排队严重的问题，需要通过资源组、队列、配额、优先级和任务 SLA 做治理。
大数据平台不是一个孤立系统。

上游可能来自：

核心数据库；
对象存储 OSS；
业务系统；
消息队列；
第三方数据源；
本地 IDC 数据库。

下游可能包括：

BI 报表；
监管系统；
风控系统；
账务系统；
AI 训练任务；
RAG 知识库；
经营分析系统。

如果只迁移表，不迁移任务和依赖关系，就会出现：

数据搬过去了，但下游无法消费；
调度跑不起来；
业务口径对不齐；
报表结果异常；
风控指标不一致。

| 校验层级               | 校验内容                 |
| ------------------ | -------------------- |
| 表级校验               | 行数、分区数、数据量大小         |
| 字段级校验              | 金额、余额、状态、时间戳、枚举值分布   |
| 主键级校验              | 主键唯一性、重复数据、缺失数据      |
| 指标级校验              | 监管指标、风控指标、业务报表结果     |
| Hash / Checksum 校验 | 对关键字段组合做一致性判断        |
| 应用层结果校验            | 验证最终报表、风控输出、账务结果是否一致 |


### 3. 目标方案设计

整体方案分为五层。

第一层是数据接入层。通过 DataWorks 数据集成能力，将本地数据库、云上数据库、对象存储以及其他业务系统的数据接入 MaxCompute。对于跨云数据传输，优先通过专线、云企业网或安全网关打通网络，保证链路带宽、稳定性和安全性。

第二层是计算与存储层。使用 MaxCompute 承载核心离线数仓，包括 ODS、DWD、DWS、ADS 等分层模型。对不同业务线、不同租户、不同敏感级别的数据进行项目空间隔离，并结合资源组、计算配额和优先级队列，保证核心监管、风控、财务类任务优先执行。

第三层是开发与调度层。通过 DataWorks 统一管理数据开发、任务编排、周期调度、补数据、失败重跑和依赖关系。迁移时先将原平台任务依赖关系梳理成 DAG，再按业务域、系统重要性和上下游影响范围进行分批迁移。

第四层是治理与安全层。针对金融合规要求，设计数据分级分类、敏感字段识别、脱敏策略、访问权限最小化、操作审计和审批流程。对生产数据访问采用角色授权和项目空间隔离，避免开发、测试、生产权限混用。

第五层是监控与运营层。建设迁移大屏和运行监控大屏，持续观察数据同步进度、任务成功率、任务耗时、资源使用率、队列等待时间、数据校验结果和异常告警。这样在迁移过程中可以把风险从“事后发现”前移到“过程可见”。

### 4. 迁移实施路径

迁移不是一次性切换，而是分阶段推进。

第一阶段是资产盘点。先盘点表、任务、调度周期、上下游依赖、数据量、分区策略、权限策略、UDF、自定义脚本、外部系统调用关系，并按照重要性分为核心链路、重要链路和普通链路。

第二阶段是依赖梳理。通过任务依赖、表血缘、业务系统调用关系，确定哪些数据链路必须整体迁移，哪些可以先双跑，哪些需要等待其他系统割接完成后再迁。

第三阶段是平台搭建与版本升级。搭建 DataWorks + MaxCompute 目标环境，同时处理版本差异问题，比如 SQL 兼容性、函数兼容性、调度参数、权限模型、资源组策略等。对于不兼容任务，提前做改造和回归测试。

第四阶段是数据迁移和任务迁移。先迁移低风险、低依赖的业务域，再迁移核心业务域。迁移过程中采用“全量同步 + 增量同步 + 双跑校验 + 最终割接”的方式，避免一次性切换带来的不可控风险。

第五阶段是数据校验。这里我会强调金融场景不能只做表级校验，而是设计多层校验：

* 表级校验：行数、分区数、数据量大小；
* 字段级校验：关键字段空值率、枚举值分布、金额字段汇总；
* 主键级校验：主键唯一性、重复数据、缺失数据；
* Hash 校验：对关键字段组合做 checksum；
* 业务口径校验：例如交易金额、账户余额、监管指标、风控指标是否一致。

第六阶段是灰度割接。对下游系统采用分批切流，先让部分报表、部分任务、部分业务域使用新平台，确认稳定后再扩大范围。最终割接时保留回滚窗口，旧平台在一段时间内不立即下线，作为风险兜底。

### 5. 风险控制设计

这个项目最大的风险不是技术跑不通，而是金融数据不一致和割接影响业务。

所以我们设计了几个风险控制点。

第一是双跑机制。核心任务在新旧平台并行运行一段时间，对比结果一致后再正式切换。

第二是分批迁移。按照业务域和风险等级迁移，不做大爆炸式迁移。

第三是回滚预案。如果割接后发现关键指标异常，可以快速把下游依赖切回旧平台，同时保留旧平台数据同步。

第四是监控大屏。迁移期间持续展示同步进度、失败任务、异常分区、校验失败表、资源瓶颈和 SLA 风险。

第五是权限和合规前置。不是等数据迁完再补权限，而是在数据入湖、开发、调度、生产访问前就设计好权限边界和审计机制。

### 6. 项目价值与结果

这个项目最终的价值主要体现在四个方面。

第一，平台能力升级。客户从分散的数据处理模式，升级到统一的大数据开发、调度、治理和运维平台。

第二，数据迁移风险可控。通过依赖梳理、双跑、字段级校验和灰度割接，降低了金融数据迁移过程中的一致性风险。

第三，安全合规能力增强。通过权限隔离、数据分级分类、审计和脱敏，满足金融客户对数据安全和监管审计的要求。

第四，为 AI 场景打基础。MaxCompute 承载了结构化数据底座，DataWorks 统一调度数据加工链路，为后续 RAG、模型训练、智能风控和经营分析提供了更稳定的数据基础。

### 面试中可以这样总结

这个项目不是单纯做一个大数据产品上线，而是围绕金融客户的数据安全、任务稳定性、上下游依赖和割接风险，设计了一套 DataWorks + MaxCompute 的云原生大数据平台迁移方案。我的重点工作是把客户的业务链路拆解成可迁移、可校验、可回滚、可监控的工程路径，确保平台升级过程中既能提升能力，又不影响金融业务连续性。


## 项目二：金融云 2AZ 到 3AZ 高可用架构迁移方案

### ——整云级容灾能力升级与业务连续性保障

这个项目的背景是，客户原有云上环境采用 2AZ 架构，能够满足一般高可用要求，但随着金融监管要求提升，以及核心业务对连续性的要求越来越高，原来的 2AZ 架构已经不能完全满足极端故障场景下的容灾目标。因此客户希望从 2AZ 演进到 3AZ，提升整体云平台的高可用能力、容灾能力和业务连续性。

我在这个项目里的角色，是从解决方案角度参与现状评估、目标架构设计、云产品迁移路径规划、网络带宽评估、风险控制和割接方案设计。

### 1. 项目背景与目标

客户是一家全球性金融机构，云上承载了大量业务系统，包括虚拟机、负载均衡、容器服务、数据库、缓存、存储、日志监控、安全组件以及大数据平台等。

原来的 2AZ 架构主要问题是，在单 AZ 故障场景下可以维持一定可用性，但如果遇到更极端的可用区级故障、容量不足、网络抖动或存储风险，恢复空间会比较有限。金融客户对 RTO、RPO、故障隔离和业务连续性要求很高，因此需要升级到 3AZ 架构。

项目目标不是简单“多加一个可用区”，而是要完成整云级高可用能力升级，包括网络、计算、存储、容器、负载均衡、数据库、中间件、监控、备份和安全策略的整体演进。

1. 环网设计价值

这个设计的价值在于：

三个 AZ 之间通过专线互联；
任意两个数据中心之间链路中断时，流量可以经由第三个 AZ 绕行；
提升跨 AZ 互通和整体可用性；
避免单链路故障导致 AZ 之间完全隔离。
2. 带宽规划

带宽规划上，可以这样讲：

初始互联带宽是 2 × 10G；
三 AZ 总互联链路可以达到 6 × 10G；
未来 ISW 的 DCI 接口可以规划升级到 100G，但需要确认运营商和设备支持能力。

### 2. 核心挑战

这个项目的难点主要有几个。

第一，迁移范围大。它不是单个系统迁移，而是整云级迁移，涉及 VM、SLB、ACK、数据库、缓存、对象存储、文件存储、日志监控、安全组、路由、DNS 等多个产品。

第二，业务连续性要求高。金融业务不能长时间停机，很多系统只能在有限窗口内切换，甚至需要做到业务无感或分钟级切换。

第三，网络带宽是关键约束。两边云环境之间通过专线互联，带宽大约是 10Gbps。这个带宽对于小规模系统迁移足够，但对于大规模 VM 镜像、磁盘、数据库、对象存储和日志数据迁移，需要精细规划迁移批次、同步窗口和限速策略，否则会影响生产流量。

第四，SLB 和入口流量需要重点设计。很多业务前面挂了 SLB，如果后端 VM 或容器已经通过 SLB 对外提供服务，那么迁移时可以通过后端服务器分批加入、健康检查、权重调整来降低连通性风险。但如果业务没有统一入口，直接依赖固定 IP、内部调用或硬编码地址，就需要额外做连接改造和 DNS 切换。

第五，存储和状态类服务风险更高。无状态计算节点迁移相对简单，但数据库、磁盘、文件存储、对象存储、缓存、消息队列这类有状态服务，需要重点关注数据一致性、同步延迟、最终割接和回滚。

### 3. 目标 3AZ 架构设计

目标架构按照“入口高可用、计算跨区、数据多副本、调度感知、监控闭环”的原则设计。

第一，网络层面。提前规划 3 个 AZ 的 VPC、交换机、路由表、安全组、ACL、NAT 网关、专线互通和云企业网路由。确保新增 AZ 不只是网络上连通，还要满足生产访问路径、运维访问路径、监控采集路径和跨云数据同步路径。

第二，入口层面。对公网或内网入口统一通过 SLB、ALB 或 NLB 承载，后端服务分布在多个 AZ。对于已有 SLB 的业务，可以通过增加第三个 AZ 的后端实例、配置健康检查、逐步调整权重实现灰度迁移。这样客户端访问入口不变，主要变更发生在后端资源池。

第三，计算层面。VM 按业务系统分组迁移。无状态 VM 优先通过镜像、快照、自动化部署或重新拉起方式迁移；有状态 VM 则需要结合磁盘同步、应用停写窗口和最终增量同步。迁移后要检查安全组、路由、主机名、IP 依赖、DNS、监控 Agent 和备份策略是否一致。

第四，容器层面。对于 ACK，需要验证 3AZ 能力，包括集群网络、节点池、Pod 调度、存储挂载、Ingress、Service、弹性伸缩和可观测性。建议新增 3AZ 节点池，通过 nodeSelector、nodeAffinity、podAntiAffinity、topologySpreadConstraints 等能力，让核心业务 Pod 分散到不同 AZ，避免所有副本集中在单个可用区。对于有状态容器，还要检查 PV、存储类和跨区挂载能力。

第五，存储层面。存储迁移可以使用 AMC 等迁移工具进行评估和执行，重点关注磁盘数据、文件数据、对象数据、同步速率、增量同步、校验和回滚。对于 PB 级数据或核心状态数据，不能只看迁移完成率，还要看数据一致性、应用读写验证和最终切换窗口。

第六，数据库和中间件层面。数据库、缓存、消息队列等需要优先确认是否支持多可用区部署、跨 AZ 副本、主备切换或三副本能力。如果产品本身支持多 AZ，则优先采用产品原生能力；如果不支持，则需要通过应用层容灾、备份恢复、只读副本或双写机制补齐。

第七，监控与安全层面。同步升级监控、日志、告警、安全组、云防火墙、KMS、IAM、审计和备份策略。3AZ 迁移不是资源搬迁完成就结束，必须确认可观测性和安全治理也覆盖第三个可用区。

### 4. 迁移实施路径

整个迁移分为六个阶段。

第一阶段是现状盘点。盘点所有云产品资源，包括 VM、SLB、ACK、数据库、缓存、存储、DNS、专线、NAT、安全组、监控、备份等。按照业务重要性、系统依赖、是否有状态、是否有 SLB、是否支持多 AZ 做分类。

第二阶段是依赖关系梳理。梳理系统之间的调用关系，包括前端入口、内部服务调用、数据库连接、对象存储访问、消息队列、日志上报、监控探针和运维入口。这里的关键是识别硬编码 IP、单 AZ 依赖、固定存储挂载和单点组件。

第三阶段是基础设施先行。先打通 3AZ 网络，包括 VPC、交换机、路由、安全组、专线和云企业网，验证 10Gbps 专线带宽下的跨云传输能力。对迁移流量和生产流量做隔离或限速，避免迁移期间把专线打满。

第四阶段是低风险资源迁移。优先迁移无状态 VM、测试环境、低风险业务和已有 SLB 保护的业务。对于有 SLB 的业务，可以把新 AZ 后端实例加入 SLB，先低权重灰度，再逐步提升权重。如果健康检查异常，可以快速摘除后端，不影响整体入口。

第五阶段是核心系统迁移。核心系统采用“预同步 + 灰度 + 双跑 + 割接 + 回滚窗口”的方式。存储类数据通过 AMC 等工具做全量迁移和增量同步，在最终割接前执行数据校验。数据库和中间件则优先使用产品自身的多 AZ 或主备复制能力，减少人工迁移风险。

第六阶段是容灾演练和验收。迁移完成后，不能只验证业务可访问，还要做 AZ 级故障演练，例如模拟单 AZ 后端下线、SLB 后端摘除、ACK 节点池不可用、数据库主备切换、存储访问异常等场景，验证 RTO/RPO 和业务恢复能力是否满足目标。

### 5. 关键产品迁移思路

对于 VM，如果前面已经接入 SLB，优先通过新 AZ 创建实例、加入 SLB 后端、健康检查通过后调整权重的方式迁移。这样入口不变，风险最小。

对于没有 SLB 的 VM，需要重点检查访问方式。如果依赖固定 IP 或点对点访问，建议先改造成域名访问或统一入口，再做迁移，否则切换时风险比较高。

对于 SLB，需要确认是否支持跨 AZ 后端、健康检查、会话保持、权重调整和证书配置同步。迁移时 SLB 是降低业务感知的关键组件。

对于 ACK，需要重点验证节点池跨 AZ、Pod 分散调度、Service 和 Ingress 连通性、存储挂载、弹性伸缩和 PDB。核心服务需要通过反亲和和拓扑分布策略，确保副本不要集中在一个 AZ。

对于存储，可以使用 AMC 做迁移和评估，但重点不是工具本身，而是迁移策略。需要设计全量同步、增量同步、最终停写窗口、数据校验和回滚策略。

对于数据库和缓存，需要确认产品是否支持多 AZ。如果支持，应使用原生多 AZ 能力；如果不支持，需要设计主备、只读副本、备份恢复或业务层降级方案。

对于日志和监控，需要确保第三个 AZ 的资源接入监控、日志采集、告警规则和大屏展示，否则迁移后会出现“资源迁过去了，但运维不可见”的问题。
1. TR 的价值

多个 VPC 只需要分别接入 TR，就可以通过中心化路由实现互通，不需要每两个 VPC 都建立 Peering。

TR 可以统一管理：

VPC-to-VPC 路由；
VPC-to-IDC 路由；
跨域连接策略；
隔离域；
安全策略；
路由传播；
路由学习。

1. 双云并行阶段

当旧云和新云同时提供服务时，可以在企业侧或本地 DNS 中配置不同后缀区的转发或委派。

例如：

ali.cloud.cn.hsbc
→ 指向旧云 Cloud DNS
shali.cloud.cn.hsbc
→ 指向新云 Cloud DNS

这样旧云和新云可以同时解析，便于双云并行和灰度验证。

2. 新云独立提供服务阶段

当迁移完成、旧云准备退役后，本地 DNS 可以把相关委派统一切到新云 Cloud DNS。

迁移后的效果是：

API 域名保持不变；
Cloud DNS 记录层面已经指向新云服务 IP；
未迁移项目继续解析到旧云；
已迁移项目解析到新云；
最终旧云退役后，统一由新云 DNS 承接解析。

### 6. 风险控制

这个项目最核心的风险控制有五点。

第一，带宽控制。因为专线带宽约 10Gbps，所以需要提前评估迁移数据量、迁移窗口和业务峰谷，避免迁移流量影响生产业务。

第二，灰度切换。优先通过 SLB 权重、DNS TTL、ACK 节点池扩容和业务分批切换实现灰度，不做一次性大切换。

第三，回滚机制。每一批迁移都要定义回滚条件，比如错误率升高、延迟升高、数据校验失败、健康检查失败、同步延迟超过阈值等。

第四，数据校验。对于存储和数据库类迁移，要做容量、文件数、对象数、checksum、业务查询结果等多维度校验。

第五，演练验收。迁移完成后必须做 AZ 级别演练，证明 3AZ 架构真的提升了可用性，而不是只是资源分布到了三个可用区。

### 7. 项目价值

这个项目的价值主要体现在三方面。

第一，提升金融级高可用能力。通过 3AZ 架构，降低单 AZ 故障对业务的影响，提升平台整体容灾能力。

第二，降低迁移风险。通过专线带宽评估、SLB 灰度、AMC 存储迁移、ACK 跨 AZ 调度、监控大屏和回滚预案，把整云迁移拆成可控步骤。

第三，提升后续云平台标准化能力。迁移完成后，客户不仅获得了 3AZ 架构，也沉淀了一套适用于 VM、容器、存储、数据库、中间件和大数据平台的迁移方法论，为后续更多系统上云、云间迁移和容灾演练提供标准模板。

### 面试中可以这样总结

这个项目的重点不是简单从 2AZ 扩到 3AZ，而是把整云架构按照金融级高可用标准重新梳理了一遍。我重点关注网络带宽、SLB 入口、VM 和 ACK 计算迁移、AMC 存储迁移、数据库和中间件多 AZ 能力、监控告警和回滚机制。通过分批迁移、灰度切换和容灾演练，确保客户在提升架构可靠性的同时，把业务中断和数据风险控制在可接受范围内。
